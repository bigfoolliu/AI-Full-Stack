# AI全栈转岗

## 1. 目标与背景

你目前是一名普通的 Python 后端开发工程师，目标岗位是 AI 全栈开发工程师。这个岗位通常要求候选人具备以下能力：

- 能完成后端服务开发
- 能接入和落地大模型能力
- 能完成常规前端页面开发与联调
- 能处理模型部署、容器化、基础运维与系统迭代

对于已有 Python 后端经验的人来说，这并不是从零转行，而更像是一次能力扩展：从“后端开发”升级为“能独立交付 AI 应用的人”。

核心判断如下：

- 你的后端基础本身就是优势，不需要推倒重来
- 真正需要补的是前端交付能力、AI 应用工程能力、以及基础部署能力
- 求职时最重要的不是学过多少概念，而是能否拿出完整项目证明你具备落地能力

一个现实且可执行的目标是：

- 用 3 到 4 个月完成转岗准备
- 以前端 Vue 为主栈
- 以后端 FastAPI 为主栈
- 做出 2 到 3 个完整 AI 项目作为作品集

---

## 2. 这个岗位到底在招什么人

从岗位描述来看，这类岗位并不是要你同时成为：

- 大模型研究员
- 资深前端架构师
- 分布式底层专家
- 云原生平台工程师

更常见的真实要求是：你要能独立做出一个 AI 应用，并把它部署运行起来。

也就是说，企业真正看重的是以下闭环能力：

1. 后端服务能写
2. 大模型 API 能接
3. RAG 与向量库会落地
4. 基础前端能完成页面与联调
5. Docker、数据库、缓存、日志、部署不陌生

这类岗位更接近以下角色：

- AI 应用工程师
- LLM 应用工程师
- AI 全栈开发工程师
- 后端偏 AI 全栈工程师

所以，你的正确定位不是“我要把所有方向都学到很深”，而是：

**以后端为主，补齐前端与 AI 工程能力，形成独立交付 AI 产品的能力。**

---

## 3. 是否适合用 Vue 作为前端主栈

答案是：可以，而且对你很合适。

如果岗位要求写的是“会使用 Vue/React 完成日常页面开发与联调”，那么你完全可以把 Vue 作为主要框架。

原因如下：

1. Vue 对后端开发者通常更友好，上手更顺
2. 企业内部后台、知识库后台、配置平台、工单系统这类页面，Vue 做起来效率很高
3. AI 全栈岗位大多并不要求你成为高级前端工程师，而是要求你能完成页面、联调、交付
4. 你可以以 Vue 为主，同时具备基本的 React 可读可改能力，这样更利于投递

最推荐的 Vue 技术组合如下：

- `Vue 3`
- `TypeScript`
- `Vite`
- `Composition API`
- `<script setup>`
- `Vue Router`
- `Pinia`
- `Element Plus`

这套组合非常适合做：

- AI 对话页面
- 知识库后台
- Prompt 配置页面
- 模型参数配置页面
- 文件上传与解析状态页面
- 检索结果展示页面
- 工单与日志后台

建议策略：

- 主力前端栈选 Vue
- React 不必深入，但最好能看懂基础组件和简单联调代码

## 4. 转岗的核心学习策略

### 4.1 不要追求“全都学精再投”

正确方式不是：

- 先把前端学完
- 再把 AI 学完
- 再把部署学完
- 最后才开始做项目和投简历

这种方式通常会导致周期很长，而且学到后面会失去方向。

更合理的方式是：

1. 先补齐最小技术闭环
2. 立刻做项目
3. 边做边学
4. 边整理简历边开始投递

### 4.2 学习顺序建议

建议按下面这个顺序推进：

1. 先补 Vue 基础与前端联调
2. 再补 FastAPI 的 AI 服务开发能力
3. 再补 RAG、向量库、Prompt、流式输出
4. 再补 Docker、Redis、PostgreSQL、Nginx 这些工程化能力
5. 用 2 到 3 个项目把这些能力串起来

### 4.3 最终目标不是“学会知识点”，而是“拥有项目证明”

判断自己是否准备好投递，不看你学了多少视频，而看你是否具备以下证据：

- 有完整的前后端项目
- 有 README 和运行说明
- 有架构设计和技术说明
- 有能讲清楚的优化点
- 有简历中的项目表述

## 5. 你需要补齐的能力地图

### 5.1 后端能力

你已有 Python 后端基础，重点升级这些能力：

- `FastAPI`
- 异步接口
- 文件上传
- 用户鉴权
- 日志与异常处理
- 任务状态管理
- Redis
- PostgreSQL
- 流式输出 `SSE` 或 `WebSocket`

### 5.2 前端能力

作为 AI 全栈岗位，前端不需要做到很深，但要能独立做页面与联调：

- Vue 组件开发
- 路由
- 状态管理
- 登录与权限
- 请求封装
- 表单与表格
- 上传与进度显示
- 聊天页面
- Markdown 渲染

### 5.3 AI 应用能力

这是岗位核心：

- Prompt 基础
- Embeddings
- RAG
- 向量库
- 文档解析
- Chunk 切分
- 检索召回
- 引用溯源
- Structured Output
- Tool Calling
- Token 成本控制
- 幻觉与稳定性处理

### 5.4 工程化与部署能力

至少要掌握：

- `Dockerfile`
- `docker-compose`
- 环境变量管理
- `Nginx`
- PostgreSQL
- Redis
- 基础日志
- 健康检查
- 简单部署流程

Kubernetes 可以了解基础概念，但转岗初期不要求非常深入。

## 6. 推荐技术栈

建议你先不要在技术选型上发散，尽量固定成一套主线。

### 6.1 前端栈

- `Vue 3`
- `TypeScript`
- `Vite`
- `Vue Router`
- `Pinia`
- `Element Plus`
- `Axios`
- `Markdown-it` 或类似 Markdown 渲染库

### 6.2 后端栈

- `Python`
- `FastAPI`
- `Pydantic`
- `SQLAlchemy` 或 `SQLModel`
- `PostgreSQL`
- `Redis`
- `Celery` 或轻量后台任务方案

### 6.3 AI 应用栈

- 大模型 API 接入
- Embedding 模型
- `Qdrant` 或 `Milvus`
- RAG 检索链路
- 文档解析工具
- Prompt 管理

### 6.4 部署与运维栈

- `Docker`
- `Docker Compose`
- `Nginx`
- Linux 基础命令
- Git

### 6.5 为什么推荐 Qdrant

对于转岗和作品集来说，`Qdrant` 是一个非常务实的选择：

- 文档清晰
- 学习曲线相对平稳
- 对语义检索、过滤、混合检索支持好
- 很适合作为个人项目和作品集的向量库

建议你做 3 个项目，至少完成前 2 个。

### 8.1 项目 1：企业知识库问答系统

推荐技术栈：

- 前端：`Vue 3 + TypeScript + Pinia + Element Plus`
- 后端：`FastAPI + PostgreSQL + Redis`
- AI：`Embeddings + RAG + Qdrant`
- 部署：`Docker Compose + Nginx`

- 登录与权限
- 知识库管理
- 文档上传
- PDF/Word/TXT 解析

- 检索问答
- 流式返回
- 引用来源展示
- 多轮会话

- 独立实现 RAG 全链路
- 支持流式输出和引用溯源
- 支持知识库管理与检索参数配置
- 完成前后端与向量库、数据库、缓存的一体化部署

### 8.2 项目 2：AI 客服 / 工单 Copilot

这是体现业务落地能力的项目。

- 展示 AI 与业务系统结合能力

## 11. 简历定位建议

### 11.1 你的定位不要写成“转行 AI”

更好的定位是：

- `Python后端工程师 / AI应用工程师（Vue 全栈方向）`
- `Python后端工程师 / AI全栈开发工程师`
- `LLM应用开发工程师`

这样会比“我想转 AI”更成熟，也更贴近岗位。

### 11.2 技能栈写法建议

推荐写成这几组：

- 后端：`Python`、`FastAPI`、`PostgreSQL`、`Redis`
- 前端：`Vue 3`、`TypeScript`、`Pinia`、`Element Plus`
- AI：`LLM 应用开发`、`RAG`、`Embeddings`、`Qdrant`
- 工程化：`Docker`、`Nginx`、`Git`、`Linux`

### 11.3 项目经历要怎么写

不要写：

- 学习了 Vue
- 学习了 RAG
- 做了一个 AI 项目

要写成结果导向、工程导向：

- 独立负责基于 `Vue 3 + FastAPI` 的企业知识库问答系统开发，完成知识库管理、文档上传解析、向量检索、流式对话、引用溯源与后台配置功能。
- 基于 `Qdrant` 搭建 RAG 检索链路，支持按知识库过滤、检索参数配置与回答反馈闭环，提升问答效果与可解释性。

- 更快知道市场反馈
- 更早发现自己的短板
- 面试过程本身也会倒逼你补能力
